## 🏗️ [Design Pattern] 디자인 패턴 (feat. 옵저버 패턴)

### 1. 디자인 패턴이란?

**Design**은 '설계'를, **Pattern**은 '반복되는 형태'를 의미합니다. 건축가들이 집을 지을 때 검증된 설계 방식을 따르듯, 소프트웨어 개발에서도 자주 발생하는 문제들을 해결하기 위해 선배 개발자들이 정리해둔 **"검증된 설계 방식"**이 바로 디자인 패턴입니다.

> 💡 **Design Pattern** = 개발자들이 오랜 경험을 통해 정리한 "이럴 땐 이렇게 만들면 좋더라"는 노하우 모음집

---

### 2. 패턴 없이 코드를 작성하면 벌어지는 일

간단한 쇼핑몰 알림 시스템을 예로 들어보겠습니다. 주문이 들어오면 고객에게 이메일을 보내는 기능입니다.

#### ❌ 문제의 시작: 수정의 연쇄 반응

처음에는 이메일만 있으면 됐지만, 점차 SMS, 카카오톡, 슬랙 알림 요구사항이 늘어납니다. 패턴 없이 코드를 짜면 다음과 같은 **'파라미터 폭발**과 **조건문 지옥**을 만나게 됩니다.

```python
class Order:
    def __init__(self, customer_email, customer_phone, kakao_id, slack_webhook, 
                 use_email, use_sms, use_kakao, use_slack):  
        # 파라미터가 끝도 없이 늘어납니다...
        self.use_email = use_email
        self.use_sms = use_sms
        # ... 생략 ...

    def place_order(self):
        print("주문이 접수되었습니다")
        
        # 새로운 알림이 추가될 때마다 이 if문을 계속 수정해야 합니다.
        if self.use_email:
            print(f"{self.customer_email}로 이메일 발송!")
        if self.use_kakao:
            print(f"{self.kakao_id}로 카카오톡 발송!")

```

**이 코드의 주요 문제점:**

* **수정의 연쇄반응:** 알림 하나 추가할 때마다 `Order` 클래스 내부를 뜯어고쳐야 합니다.
* **책임의 비대화:** `Order` 클래스가 주문도 처리하고, 이메일/SMS 로직까지 전부 알아야 합니다.
* **재사용 불가:** SMS 발송 로직만 따로 떼서 다른 곳에 쓰기가 매우 어렵습니다.

### 3. 해결사 등장: 옵저버 패턴 (Observer Pattern)

이 문제를 해결하기 위해 **옵저버 패턴**을 적용해 보겠습니다. 핵심 아이디어는 **"알림을 보내는 주체(Order)와 알림을 받는 수단(Notifier)을 분리하는 것"** 입니다.

#### ✅ 패턴 적용 코드

```python
# Step 1: 모든 알림 방식이 따를 규칙(인터페이스) 정의
class Notifier:
    def send(self, message):
        pass

# Step 2: 각 알림 방식을 독립적인 클래스로 분리 (확장이 쉬워짐!)
class EmailNotifier(Notifier):
    def send(self, message):
        print(f"📧 이메일 발송: {message}")

class KakaoNotifier(Notifier):
    def send(self, message):
        print(f"💬 카카오톡 발송: {message}")

# Step 3: Order 클래스는 이제 알림 방식의 세부 내용을 몰라도 됩니다.
class Order:
    def __init__(self):
        self.notifiers = []  # 알림 목록만 관리
    
    def add_notifier(self, notifier):
        self.notifiers.append(notifier)
    
    def place_order(self):
        print("주문이 접수되었습니다")
        for notifier in self.notifiers:
            notifier.send("주문이 완료되었습니다!")

```

#### ✨ 패턴 적용 후 달라진 점

1. **확장 용이:** 새로운 알림(예: 텔레그램)이 추가되어도 `Order` 클래스는 건드릴 필요가 없습니다.
2. **테스트 쉬움:** `EmailNotifier`만 따로 떼서 테스트하기 좋습니다.
3. **책임 분리:** `Order`는 주문에만 집중하고, 각 알림 객체는 자기 발송 로직에만 집중합니다.

### 4. 한눈에 비교하는 흐름의 변화

디자인 패턴을 적용하기 전과 후의 로직 흐름을 비교해 보세요.

| 구분 | 패턴 미적용 (절차적) | 패턴 적용 (옵저버 패턴) |
| --- | --- | --- |
| **주체** | 주문 시스템이 모든 발송 로직을 수행 | 주문 시스템은 '이벤트'만 알림 |
| **의존성** | 높은 의존성 (수정 시 위험함) | 낮은 의존성 (자유로운 추가/삭제) |
| **구조** | 내부 코드 수정 필요 | 외부에서 알림 객체 주입 가능 |

---

### 5. 마무리하며: 자주 쓰이는 디자인 패턴들

오늘 살펴본 옵저버 패턴 외에도 개발 생산성을 높여주는 다양한 패턴들이 있습니다.

* **Singleton:** 프로그램 전체에서 객체를 딱 하나만 유지하고 싶을 때
* **Factory:** 객체 생성을 전담하는 공장을 만들고 싶을 때
* **Strategy:** 실행 중에 알고리즘(전략)을 자유롭게 교체하고 싶을 때
* **Decorator:** 기존 코드 수정 없이 기능을 계속 덧붙이고 싶을 때
